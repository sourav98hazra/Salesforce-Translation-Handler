# Foundation fixes: backend wiring, validation, audit, counters, BOM, secrets, CI

Nine targeted fixes that shore up the translation pipeline's correctness and operability. The backend selection was previously dead: `TranslationWorker` hardcoded `GoogleFreeTranslator()` regardless of the user's settings choice. That's now wired through a centralized factory. Alongside that, whitespace-only validation was unreachable dead code, audit columns were incomplete, failure/cancellation didn't count toward skipped totals, and API keys were persisted in plaintext QSettings. All nine are resolved with good test coverage (42 new tests across 9 files).

**Watch for:** The `check_backend_available` function checks the env var but doesn't know about keys stored in keyring, creating a possible false-negative at settings-dialog load time before the user types a key (confirmed, minor UX issue). The keyring fallback path silently swallows exceptions without telling the user their key wasn't persisted (confirmed).

## High-level view

The backend factory (`translate/factory.py`) centralizes construction, availability-checking, and metadata for all four translator backends. `TranslationWorker` and the CLI both route through `make_backend(key, **kwargs)`, ending the hardcoded Google-only path. The factory's `check_backend_available` drives a live status indicator in the settings dialog.

Secrets management moves API keys out of QSettings into OS credential stores via `keyring`, with env-var fallback. The settings dialog gates persistence behind an explicit "Remember API key" checkbox and the key field uses password-echo mode. A `.ini` file on disk no longer contains auth material.

The validation fix restructures `_check_entry` so the whitespace-only check executes before the early return â€” previously it was dead code. The CLI validate command now exits non-zero on errors, making it CI-friendly.

The runner's `_mark_failed` and `_mark_cancelled` now increment `skipped_rows` and `self._skipped` inside the completion lock, closing an accounting gap where failures disappeared from totals. The progress denominator guard in `phase3_translate.py` prevents `ZeroDivisionError` when scope excludes all rows.

The STF parser switches to `utf-8-sig` encoding and strips BOM in `parse_stf_text`, plus normalizes stray CR characters â€” handles files from Windows Salesforce exports and older editors that inject BOM.

CI is a straightforward GitHub Actions matrix (Python 3.10-3.12) with system dependencies for headless Qt, coverage upload for 3.11 only.

## Backend factory and `check_backend_available` gap

`check_backend_available` performs two checks: SDK importability and env-var presence. It doesn't consult keyring. The settings dialog works around this by short-circuiting the status check when the user has typed something into the API key field (`_update_backend_status` checks `has_local_key`). But on initial load, `_load_values` triggers the following ordering issue:

```python
backend = gui_settings.get_str(gui_settings.KEYS.backend, "google")
for i in range(...):
    if self._backend_combo.itemData(i) == backend:
        self._backend_combo.setCurrentIndex(i)  # triggers _update_backend_status with empty field
        break
# ... then later ...
stored_key = gui_secrets.retrieve_api_key(backend) or ""
self._api_key_field.setText(stored_key)  # triggers _update_backend_status again, now green
```

`setCurrentIndex` fires `_update_backend_status` with an empty field, flashing a red status momentarily until `setText` corrects it. Minor UX flicker on dialog open for keyring users.

## Silent keyring storage failure

`store_api_key` catches all exceptions and returns `False`, but `settings_dialog._save_values` ignores the return:

```python
if self._remember_key_check.isChecked() and api_key:
    gui_secrets.store_api_key(backend, api_key)  # return value ignored
    gui_settings.set_str(f"translation/remember_api_key_{backend}", "1")
```

If storage fails (permissions, locked keychain, D-Bus timeout), the remember-flag is still set to "1". Next launch: code tries to retrieve from keyring, gets nothing, field is empty. The user's key is silently lost.

## Progress counter drift

`_total_rows` represents scope estimate (in-scope rows only). The runner emits progress events for *every* row including out-of-scope ones (via `_fill` in `_classify`). The `row_translated` signal fires for status "Translated", "Skipped", "Cancelled", "Fallback" â€” all of which increment `_current_row`. If a 1000-row document has a scope selecting 200 rows, `_total_rows` is 200 but `_current_row` reaches 1000. The intermittent percentage computation `(self._current_row * 100 // self._total_rows)` yields values well over 100%.

## Runner failure/cancellation accounting

Both `_mark_failed` and `_mark_cancelled` now increment `summary.skipped_rows += 1` and `self._skipped += 1` inside `self._completion_lock`. Previously they filled the status slot without touching counters, violating the invariant `translated_count + skipped_count == len(entries)`. The gap-prevention sweep at `run()` end didn't catch these because their status slot was already non-None.

## CI and test gaps

No linting step (ruff) is included in the workflow despite ruff being mentioned in contribution requirements.

Not tested: keyring store/retrieve integration (mocking keyring would be valuable since it's now a critical path for key persistence), the `sanitize_error_message` regex, the `_update_backend_status` ordering race, and multi-language batch translation in the CLI.
