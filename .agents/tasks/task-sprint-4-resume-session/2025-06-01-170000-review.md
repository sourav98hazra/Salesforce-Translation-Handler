# Checkpoint persistence and session restore for resumable translations

Sprint 4 adds two persistence layers: a per-row checkpoint that survives crashes mid-translation, and a full session project file (.stxproj) that restores the entire app state across launches. The checkpoint integrates at the runner's `_fill_translated` boundary — each completed row is atomically flushed to a JSON file keyed by source-path + target-language, and loaded back into the classify loop on resume so checkpointed rows skip both the TM and the translator backend entirely. The session layer serializes the full AppState (document entries, language settings, phase status, translation audit data, undo commands) with a versioned JSON format and SHA-256 path-based isolation.

Watch for: **concurrent checkpoint writes are unguarded** (confirmed) — `save_progress` mutates a shared dict and performs file I/O from parallel worker threads without synchronization; **failed/fallback rows are never checkpointed** (confirmed), meaning a crash after a transient error leaves that row in a permanent retry gap; **session restore doesn't verify source file still exists** (confirmed) — restoring a session for a moved/deleted file leaves `source_stf_path` pointing at nothing.

## High-level view

The checkpoint store does a full JSON rewrite on every row completion, with atomic rename for crash safety. The run ID is a truncated SHA-256 of `source_file::target_lang`. The lack of thread synchronization on the in-memory `_loaded` dict and the `_write()` call is a concurrency defect when the runner operates with `workers > 1`.

The runner integration is well-placed: checkpointed rows are recognized in `_classify()` before any TM or translator work begins, achieving the "no re-billing" goal. However, only successfully-translated rows pass through `_fill_translated` and get checkpointed. Rows that hit `_mark_failed` are never persisted, so a crash-and-resume cycle will re-attempt them — arguably correct for transient errors but the checkpoint gives no visibility into which rows were attempted-and-failed versus never-attempted.

The session undo stack integration relies on a private `_undo_commands` attribute — a coupling that works today but is fragile if AppState's undo representation changes. Only the commands list is serialized, not the redo pointer position.

<summary>Issues (6)</summary>

1. **Race condition in concurrent checkpoint writes** (confirmed) — `save_progress()` is called outside `_completion_lock` from parallel worker threads. Multiple threads can interleave reads/writes to `self._loaded` dict and the `_write()` file I/O, risking data loss or corrupt JSON. Add a threading lock inside `CheckpointStore` or call `save_progress` under the existing `_completion_lock`.
2. **Failed rows not checkpointed** (confirmed) — Rows that go through `_mark_failed` (transient errors, empty results, glossary token loss) are never written to the checkpoint. On resume, they are re-classified as "needs translation" and re-attempted. Permanent failures (e.g., "no change" where source == translation) will retry indefinitely.
3. **Session restore with missing source file** (confirmed) — `_apply_session_data` never checks that `source_file_path` still exists on disk. If the source was moved/deleted, the app holds a stale `source_stf_path` and subsequent operations that re-read it will fail.
4. **Session undo stack coupling via private attribute** (confirmed) — `SessionManager.save()` accesses `state._undo_commands` via `hasattr`. If AppState refactors its undo representation, serialization silently drops the undo history with no error. Additionally, only commands are serialized, not the `_index` pointer — on restore the undo/redo split position is lost.
5. **Checkpoint written on dedup-replay rows** (confirmed) — In `_run_parallel`, dedup duplicates call `_fill_translated` which calls `save_progress`, writing individual checkpoint entries for rows that were never independently translated. Benign for correctness but inflates checkpoint I/O proportional to duplicate count.
6. **CLI `--reset-checkpoint` semantics unclear** (likely) — If a user passes `--reset-checkpoint` without `--no-resume`, the checkpoint is cleared and then immediately re-created row by row. The behavior is technically correct (fresh run with persistence) but the flag name suggests "disable checkpoints entirely", which is actually `--no-resume`.

## Concurrent access to CheckpointStore from parallel workers

The runner calls `checkpoint.save_progress(index, entry.key, translation, status)` immediately after releasing `self._completion_lock` inside `_fill_translated()` (runner.py ~line 788). When `workers > 1`, multiple threads can concurrently:

1. Mutate `self._loaded` (a plain `dict`).
2. Call `self._write()`, which reads `self._loaded` to serialize JSON and performs file I/O with `tempfile.mkstemp` + `os.replace`.

Two threads racing through `_write()` will each create a temp file, serialize their own snapshot of `_loaded` (which may or may not include the other thread's just-added entry), and then both call `os.replace`. The last writer wins, potentially dropping an entry that the other thread just recorded.

```
Thread A: save_progress(5, ...) -> _loaded[5] = {...} -> _write() -> mkstemp -> json.dump -> os.replace
Thread B: save_progress(7, ...) -> _loaded[7] = {...} -> _write() -> mkstemp -> json.dump -> os.replace
```

If Thread B's `json.dump` captures `_loaded` before Thread A's `_loaded[5] = {...}` executes, the final file on disk is missing entry 5. On resume, row 5 would be re-translated — violating the "no re-billing" guarantee for that row.

Fix: add a `threading.Lock` inside `CheckpointStore` guarding `save_progress` and `_write`, or move the `save_progress` call inside the existing `_completion_lock` in the runner.

## Failed rows and the checkpoint gap

`_mark_failed` records the row as "Skipped" in `_statuses` and `_new_entries` but never calls `checkpoint.save_progress`. This means:

- Row hits a transient 429/500 from the translator API -> marked "Fallback to original (RequestError)" -> not checkpointed.
- App crashes 10 rows later -> on resume, that failed row is re-classified as "needs translation" (it has no translation, no checkpoint entry) -> translator is consulted again.

For transient errors this retry-on-resume behavior is reasonable. But permanent failures (e.g., "Fallback to original (no change)" where the translator returns the source text verbatim) will be retried on every resume indefinitely. Consider checkpointing permanent-failure rows with a distinct status so they aren't re-attempted.

## Checkpoint I/O scaling

Every call to `save_progress` rewrites the entire checkpoint JSON. For a 10,000-row document, this means 10,000 atomic write cycles each serializing an increasingly large JSON object. Total I/O is O(n^2) over the run. For typical documents (< 5k rows, each write < 500KB) this won't be a bottleneck. For very large multilingual orgs with 30k+ rows per language, it could become noticeable — consider batching writes (e.g., flush every N rows or every T seconds) behind the existing lock if one is added.

## Session restore with missing source file

`_try_restore_session` in `MainWindow.__init__` reads the most recent file from settings, checks `has_session()`, loads the session data, and calls `_apply_session_data`. At no point does it verify `Path(source_file_path).is_file()`. If the user moved or renamed the source STF between sessions, the restored state holds a broken `source_stf_path`. Phase 1 (Import) would show "file loaded" status but any re-parse or re-export would fail with a confusing error.

Fix: check `Path(source_path).is_file()` in `_try_restore_session` before applying, and fall back to clearing the stale session.

## Session serialization gaps

Missing from the serialized format:
- `pending_edits` (mentioned in the FEAT-002 spec but absent from implementation) — unsaved in-table edits that haven't been committed to the Document would be lost on session save.
- The undo stack `_index` (redo pointer position) — only the commands list is serialized. On restore, the redo position is unknown, so the app cannot correctly determine which commands are undone vs. redone.

## Test coverage gaps

Not tested:
- Concurrent `save_progress` from multiple threads (would expose the race condition described above).
- Session restore when the source file no longer exists on disk.
- GUI `_on_reset_checkpoint` button behavior (no pytest-qt test for Phase 3 checkpoint integration).
- CLI `stx session info` and `stx session reset` commands (no CLI integration tests).
- Resume with a checkpoint written by a hypothetical future version of the code (forward-compatibility — current code returns `{}` for unknown versions, which is correct, but untested).
