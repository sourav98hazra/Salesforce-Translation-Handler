# Fuzzy translation memory via rapidfuzz (v2)

This change extends the translation memory to find approximate matches via rapidfuzz's WRatio scorer, auto-accept high-confidence fuzzy hits without network translation, and expose threshold/max-results/auto-accept knobs through the CLI, GUI settings, and GUI live feed. All five issues from the v1 review have been addressed: the TM candidate list is cached per-run, `translate_document_multi` forwards fuzzy params, below-auto-accept suggestions are logged via `LOGGER.info`, the GUI uses a structured `from_fuzzy` boolean instead of string matching, and the duplicate-source collision now keeps the first translation encountered.

**Watch for:** Race condition on `_fuzzy_candidates` initialization in parallel mode (likely) — benign but wasteful under concurrency. No new blocking concerns remain.

**Verdict**: APPROVED

## High-level view

The per-run caching fix stores the TM candidate list in `self._fuzzy_candidates` and passes it to `fuzzy_search()` via the new `candidates` parameter, eliminating per-row SQLite round-trips. The lazy-initialization pattern (check-then-set without a lock) means multiple threads in parallel mode can each trigger `all_sources()` before the first write lands, but the result is idempotent so correctness is preserved.

`translate_document_multi` now accepts and forwards `fuzzy_threshold`, `fuzzy_max_results`, and `fuzzy_auto_accept_threshold`, passing them directly to `translate_document` for each target language.

Below-auto-accept fuzzy matches now emit a `LOGGER.info` message containing the query, matched source, score, and the auto-accept threshold. This surfaces suggestions in log output without requiring a new event type.

The GUI no longer parses the status string for "fuzzy". `TranslationProgress` gained a `from_fuzzy: bool` field that propagates through `_emit_progress` → `TranslationWorker.row_translated` (now a 4-arg signal) → `_on_row_translated`, cleanly decoupling the display logic from status text.

The `FuzzyMatcher.find_matches` dict comprehension was replaced with a first-wins insertion loop (`if src not in source_to_translation`), making duplicate-source behavior deterministic and documented in the code comment.

## Fuzzy candidate cache race under concurrency

The fix for the v1 per-row full-table scan caches candidates lazily:

```python
if self._fuzzy_candidates is None:
    self._fuzzy_candidates = self.memory.all_sources(
        self.source_lang, self.target_lang
    )
```

This runs inside `_translate_one`, which is dispatched to a `ThreadPoolExecutor` when `workers > 1`. With 4 workers and (say) 4 primary indices all submitted simultaneously, each thread can observe `None` and call `all_sources()` before any thread writes back the result. Since `all_sources()` is a read-only SELECT and all threads produce the same list, correctness is unaffected — the final value of `_fuzzy_candidates` is always valid. The cost is O(workers) redundant SQLite queries on the very first batch, typically negligible relative to the network translation calls that follow.

A minimal fix would be:

```python
with self._dedup_lock:
    if self._fuzzy_candidates is None:
        self._fuzzy_candidates = self.memory.all_sources(
            self.source_lang, self.target_lang
        )
    candidates = self._fuzzy_candidates
```

This is non-blocking to raise but not worth holding the PR over.
