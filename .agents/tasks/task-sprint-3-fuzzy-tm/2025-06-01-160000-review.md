# Fuzzy translation memory via rapidfuzz

The change extends the existing exact-match translation memory to return ranked similar matches using rapidfuzz's WRatio scorer. A new `FuzzyMatcher` class drives the scoring, `TranslationMemory` gains `all_sources()` and `fuzzy_search()` methods, the runner auto-accepts matches above a configurable threshold (skipping the network translator), and the CLI/GUI expose three new knobs: threshold, max-results, and auto-accept score. Sixteen new tests cover the matcher, TM integration, and runner integration.

**Watch for:** Per-row full-table scan of the TM (confirmed) will degrade on large databases. `translate_document_multi` does not forward fuzzy params (confirmed), so batch multi-language translations silently skip fuzzy matching. The GUI detects fuzzy status via substring match on the status string (confirmed), which is fragile.

## High-level view

`FuzzyMatcher` wraps `rapidfuzz.process.extract` with a `score_cutoff` and limit, returning `FuzzyMatch` dataclasses. `TranslationMemory.fuzzy_search` loads all source/translation pairs for a language pair into memory and delegates to the matcher -- but this load happens on every call, and the runner calls it once per untranslated row that misses both exact TM and dedup caches. This is the primary scaling concern.

The runner auto-accepts when the best fuzzy match scores at or above `fuzzy_auto_accept_threshold`, inserting the result into the dedup cache (preventing repeated fuzzy lookups for duplicate labels). Matches that score between threshold and auto-accept are silently dropped -- they are not surfaced as suggestions despite the feature spec and GUI log implying they would be.

`translate_document_multi` does not accept or forward fuzzy parameters, so multi-target batch translations silently skip fuzzy matching entirely.

## Per-row TM scan and performance ceiling

`TranslationMemory.fuzzy_search()` executes `all_sources()` on every call, which runs `SELECT source, translation FROM translations WHERE source_lang = ? AND target_lang = ?` and materializes the full result into a Python list of tuples. The runner invokes this inside `_translate_one`, which is called once per entry that missed both the exact TM lookup and the in-run dedup cache.

For a TM with N entries and a document with M untranslated rows (after exact-hit and dedup filtering), the cost is O(M * N) string comparisons -- rapidfuzz is fast in C, but the SQLite I/O and Python list allocation happen M times. A 10k-entry TM with a 3k-row document would allocate ~30M tuples across the run.

The fix is straightforward: load `all_sources()` once at runner construction time (or lazily on first fuzzy call) and reuse the candidate list. The candidate list is already stale-safe because the runner only *adds* to the TM during the run (via `put()`), and new entries are also inserted into the dedup cache, so they'd be caught before reaching fuzzy lookup again anyway.

```python
# In _Runner.__init__:
self._fuzzy_candidates: Optional[List[Tuple[str, str]]] = None

# In _translate_one, before fuzzy_search:
if self._fuzzy_candidates is None:
    self._fuzzy_candidates = self.memory.all_sources(self.source_lang, self.target_lang)
```

## `translate_document_multi` does not forward fuzzy params

```python
def translate_document_multi(
    ...,
    workers: int = DEFAULT_WORKERS,
    rate_limit_per_second: Optional[float] = 8.0,
) -> MultiTargetResult:
```

It calls `translate_document` without `fuzzy_threshold`, `fuzzy_max_results`, or `fuzzy_auto_accept_threshold`. Since these default to `None` / 5 / 90.0 in `translate_document`, fuzzy matching is disabled (threshold=None) for all multi-target batch runs. The CLI `run_pipeline` command uses `translate_document` directly (not `_multi`), so it's unaffected today, but any future caller using the multi path will silently lose fuzzy matching.

## Below-threshold fuzzy matches are discarded

The runner's fuzzy block:

```python
if fuzzy_matches and fuzzy_matches[0].score >= self.fuzzy_auto_accept_threshold:
    # ... use it
    return
```

When the best match scores below auto-accept but above threshold, execution falls through to network translation without logging the fuzzy suggestions. The FEAT-002 spec says "matches between threshold and auto-accept are logged as suggestions but the translator is still called," and the GUI log has a `[FUZZY]` prefix for fuzzy statuses, but there is no code path that emits a suggestion event. The suggestions simply vanish.

## GUI fuzzy-status detection is string-based

```python
if "fuzzy" in status.lower():
    fuzzy_prefix = "[FUZZY] "
```

This works today because the only status containing "fuzzy" is `"Translated (fuzzy TM)"`. A boolean `from_fuzzy` field on `TranslationProgress` (analogous to the `from_fuzzy` kwarg already on `_fill_translated`) would be robust against future status string changes.

## Test coverage

Not tested: the per-row performance path (no test exercises fuzzy on a multi-entry document to verify dedup interaction with fuzzy), the GUI settings round-trip (reading/writing fuzzy settings from QSettings), the `[FUZZY]` prefix logic in the live log, or the `translate_document_multi` gap.
