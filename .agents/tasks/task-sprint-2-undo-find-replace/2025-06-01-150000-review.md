# Undo/Redo and Find & Replace for Phase 4 editor

Two capabilities added to the translation review pipeline: a cell-level undo stack that tracks translation and approved-status edits, and a global find-and-replace system exposed via both a modal dialog and a CLI command. The undo stack is a lightweight index-pointer implementation (`UndoStack` as QObject emitting a signal on mutation) wired into `_EntriesModel.setData()` with a guard flag to prevent recursion during replay. Find & replace is factored into a shared utility module consumed by both the GUI dialog and `stx replace`. 50 new tests cover the feature surface.

**Watch for:** The GUI find-replace silently drops non-translation replacements (confirmed). The undo stack has no size cap, which could become a memory concern for long editing sessions on large documents (likely). The `_refresh_undo_actions` callback does not re-fire when the user navigates away from Phase 4 via sidebar, leaving menu actions enabled until the next stack mutation (confirmed).

## High-level view

The undo stack is a simple index-pointer list of command dataclasses, avoiding Qt's `QUndoStack` (which carries widget assumptions) in favour of a plain QObject with a `stack_changed` signal. The `_applying_undo` flag on the model prevents infinite recursion when undo/redo calls back into `setData`, introducing a coupling between the model and the stack lifecycle — if the flag is ever left set (e.g., an exception in `setData`), all subsequent edits silently skip undo tracking.

Find & replace is separated into `src/stx/find_replace.py` (pure computation: `compute_replacements`) and a GUI integration path that feeds results through `setData` one-by-one so each change becomes an undo command. The CLI calls `compute_replacements` + `apply_replacements` directly since it has no undo context. The dialog allows all four scope options (translation/label/key/all) but the GUI application loop only handles `rep.field == "translation"`, meaning non-translation scopes show a preview count then silently do nothing on Replace All.

Menu integration connects `Ctrl+Z`, `Ctrl+Y`, and `Ctrl+H` to Phase 4 via delegation methods on `MainWindow`. The undo actions are enabled/disabled by `stack_changed` but not by page-switch events, so they can remain stale after navigating away from Phase 4.

1. **GUI Replace drops non-translation fields** — `_on_find_replace` iterates replacements but only applies those where `rep.field == "translation"`. The status message reports the full count regardless. Either restrict the dialog scope to translation-only, or implement `setData` paths for label/key columns.
2. **Undo stack unbounded growth** — No size cap on `self._commands`. A single Replace All on 2,000 rows pushes 2,000 `UndoCommand` objects holding old+new strings. Add a max-depth (200-500 commands) with oldest-first eviction.
3. **Menu actions stale after page navigation** — `_refresh_undo_actions` fires on `stack_changed` but not on page-switch. Wire `self._stack.currentChanged` to also call `_refresh_undo_actions`.
4. **"Apply to all rows" does substring replacement, not full-field** — `compute_replacements` uses `re.compile(re.escape(find))` which matches substrings. If old translation is "Hello" and another row has "Hello World", that row gets partially replaced. The feature name implies whole-field semantics. Use anchored regex (`^...$`) or compare full field equality.
5. **`_applying_undo` flag not exception-safe** — If `setData` raises during undo replay, the flag stays `True` and all future edits skip undo tracking. Wrap the `setData` call in a try/finally (already done in `_apply_command_value` — confirmed safe there, but worth noting the pattern's fragility if the guard is ever used elsewhere).

## GUI replace silently drops non-translation scope matches

In `_on_find_replace`, the replacement loop filters with `if rep.field == "translation"`:

```python
for rep in replacements:
    if rep.field == "translation":
        idx = self._model.index(rep.row, _TRANSLATION_COL)
        self._model.setData(idx, rep.new_value, Qt.ItemDataRole.EditRole)
```

The dialog allows the user to select "Labels", "Keys", or "All Fields" scope. If the user picks "Labels" scope, the dialog shows a preview count of N matches, the user clicks Replace All, and the status bar reports "Replaced N occurrence(s)" — but zero actual changes are applied because label replacements have `rep.field == "label"` which is skipped. The count message is computed from `len(replacements)` (the unfiltered list), so it lies about what happened.

This is a user-facing data-integrity concern: the UI reports success while doing nothing. Either restrict the scope combo in the dialog to "Translations Only" (removing the options that can't work through the model), or extend `_on_find_replace` to handle label and key mutations by directly modifying `doc.entries[row]` and emitting appropriate signals.

## Unbounded undo stack

```python
def push(self, command: UndoCommand) -> None:
    self._commands = self._commands[: self._index]
    self._commands.append(command)
    self._index += 1
```

For a document with 5,000 entries, a single Replace All touching 2,000 rows pushes 2,000 `UndoCommand` objects. Each holds old and new string values, so memory consumption scales linearly with total characters touched. Repeating bulk operations in a long session grows the list without bound. Worth capping at 200-500 commands with oldest-first eviction.

## Menu action state drift on page navigation

`_refresh_undo_actions` checks `self._stack.currentIndex() == 3` to gate enablement, but it's only called when `stack_changed` fires. If the user navigates away from Phase 4 without touching the stack, Edit > Undo/Redo remain enabled. Pressing Ctrl+Z is harmless (the delegation short-circuits on the page check), but the enabled state is misleading. Connect `self._stack.currentChanged` to `_refresh_undo_actions`.

## "Apply to all rows" substring semantics

The inline editor's "Apply to all rows" calls `compute_replacements` with `case_sensitive=True, use_regex=False`. Under the hood, `_compile_pattern` does `re.compile(re.escape(find))` — a *substring* match. If the old translation is "Hello" and another row has "Hello World", that row gets changed to "<new_text> World" rather than being left alone.

The feature name "Apply to all rows" implies full-field replacement (replace entries that have this exact translation), but the implementation does substring replacement (replace the substring wherever it appears). For short, common translation values this could produce surprising partial edits across unrelated rows.
