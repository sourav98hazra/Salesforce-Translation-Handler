# Auto-detect source language and Windows installer infrastructure

Two unrelated features land in one sprint: a language detection module wired into all three entry points (CLI `translate`, CLI `run`, GUI phases 1 and 2), and a complete Windows installer pipeline (Inno Setup script, build orchestrator, signing script, CI workflow, documentation). The detection module samples document labels via `langdetect`, aggregates per-string detection results, and maps ISO 639-1 codes to Salesforce-specific codes. The installer work is infrastructure-only — scripts and CI config that produce a distributable `.exe` on release branches.

**Watch for:** The `translate` command parses the entire Excel workbook twice — once for detection and once for translation — which doubles I/O on large files (**confirmed**). The `sys.argv` sniffing used to determine whether `--source` was explicitly provided is fragile and will misfire if `-s` appears as a substring of another argument or is used for a different purpose in a future subcommand (**likely**). The signing workflow uses an HTTP (not HTTPS) timestamp URL, which is standard for RFC 3161 but means the timestamp request and response transit the network unencrypted (**confirmed**).

## High-level view

The detection module is a pure-function core (`detect_source_language`) that aggregates per-sample `langdetect` results into normalized confidence scores, paired with a mapping function that converts ISO 639-1 to Salesforce codes. The 60% confidence threshold mentioned in the feature spec is not enforced — the function always returns the top candidates regardless of absolute confidence, and callers apply the result unconditionally. Low-confidence guesses will silently override the default source language.

CLI integration happens in both `translate` and `run` commands via identical code blocks that inspect `sys.argv` raw to determine if `--source` was explicitly given — a brittle heuristic that bypasses Typer's own parameter resolution. In the `translate` command specifically, the workbook is loaded once for detection and then loaded again for translation, doubling file I/O.

The GUI integration is lightweight: Phase 1 runs detection after parsing and populates new "Source language" fields; Phase 2 runs detection when loading an existing workbook. Both run synchronously on the main thread, diverging from the codebase's pattern of delegating heavy work to QThread workers.

The installer pipeline is entirely Windows-specific infrastructure. The Inno Setup script, build orchestrator, signing script, and CI workflow form a complete chain from PyInstaller output to signed distributable. Version is hardcoded in both `.iss` and `pyproject.toml` with no synchronization mechanism between them.

## Double parse and `sys.argv` sniffing in the CLI

The `translate` command loads the entire Excel workbook to extract labels for detection (`doc_for_detect = import_document_from_excel(...)` at line ~270) and then loads it again a few lines later (`doc = import_document_from_excel(...)`) for the actual translation run. For a workbook with tens of thousands of rows, this doubles the openpyxl parsing time. The `run` command avoids this because it uses `parse_stf` (fast) for detection and then the same parsed document proceeds through the pipeline — the asymmetry suggests the `translate` command's approach was an oversight rather than a deliberate choice.

The check `any(arg in sys.argv for arg in ("--source", "-s"))` tests whether the literal strings `--source` or `-s` appear anywhere in `sys.argv`. If a future argument value happened to equal `"-s"`, or if the application is invoked programmatically with a modified argv, the heuristic breaks. Typer/Click expose a `Context.get_parameter_source()` method that reliably distinguishes default values from explicit user input.

## Detection accuracy and threshold behavior

`detect_source_language` normalizes accumulated scores so they sum to 1.0 across all detected languages. The top result's confidence represents its share of the total detection weight, not an absolute confidence measure. For a document whose labels are 52% Spanish and 48% Portuguese, the function returns Spanish at 0.52 confidence and the caller applies it without hesitation — there's no threshold check at either the library level or the call sites.

The `map_detected_to_salesforce` function maps `"pt"` to `"pt_BR"` unconditionally. Since `pt_PT` is also a valid Salesforce locale, this creates an implicit bias toward Brazilian Portuguese that should at minimum be documented.

## GUI thread-safety concern

Both `_detect_source_language` methods in Phase 1 and Phase 2 run on the main/GUI thread. The langdetect library is CPU-bound (n-gram profile matching), and while sampling caps at 50 strings, each `detect_langs` call builds a full language profile. This differs from the pattern used elsewhere in this codebase where I/O and CPU-heavy work (parsing STF, exporting Excel) is delegated to `QThread`-based workers. For typical Salesforce translation files the latency is probably acceptable, but it's a latent jank source for unusually large documents.

## Installer signing and credential exposure

The signing script reads `SIGN_CERT_PASSWORD` from an environment variable and passes it to signtool via command-line argument (`/p "$CertPassword"`). On Windows, command-line arguments of running processes are visible to other processes running as the same user (via `wmic` or Process Explorer). This is standard practice for CI signing, but the documentation should note that the certificate password should be ephemeral or protected by a hardware security module in production signing scenarios.

## Version synchronization gap

The `.iss` script hardcodes `#define MyAppVersion "1.5.0"` and `pyproject.toml` declares `version = "1.5.0"`. The build orchestrator doesn't inject the version from `pyproject.toml` into the `.iss` script, so a version bump in one location but not the other will produce a mismatched installer. Consider having `build_installer.py` read the version from `pyproject.toml` and pass it as a `/D` define to ISCC.

## Test coverage gaps

Not tested: CLI integration of `--detect-source` (no test invokes the CLI with this flag and verifies the source language was overridden); GUI detection behavior (no `pytest-qt` test simulates a document load and checks the source language fields); the `to_google_code` interaction (detection produces a Salesforce code which is then converted to a Google Translate code — this chain isn't tested end-to-end); the build orchestrator's subprocess invocation logic (no mocked `subprocess.run` tests, so regressions in control flow won't be caught on Linux CI).

The `test_importerror_graceful` test patches `builtins.__import__` directly — a brittle approach that may break across Python versions. A simpler strategy would be to patch `langdetect` at the module level within `stx.lang_detect` after it's been imported.
