# Auto-detect source language and Windows installer infrastructure (v2)

Two features in one sprint: a `langdetect`-based source language detection module integrated into CLI and GUI, and a Windows installer pipeline (Inno Setup, build orchestrator, signing script, CI workflow). The v1 review flagged five issues; three have been fixed (double Excel parse, `sys.argv` sniffing, missing confidence threshold). Two remain unchanged at low severity (HTTP timestamp URL, GUI synchronous detection).

**Watch for:** The signing script passes the certificate password as a command-line argument visible to co-resident processes (**confirmed**, acceptable for CI but worth documenting). Version is hardcoded in both `.iss` and `pyproject.toml` with no synchronization mechanism (**confirmed**). The `pt` -> `pt_BR` mapping is a silent regional bias that could surprise Portuguese-locale users (**confirmed**, documentation gap).

**Verdict**: APPROVED

## High-level view

The v1 double-parse issue is resolved: the `translate` command stores the parsed document from the detection pass and reuses it for translation via `if doc is None: doc = import_document_from_excel(...)`. The `sys.argv` sniffing is replaced by a sentinel default value (`_SOURCE_SENTINEL = "__auto__"`) — the code checks `source != _SOURCE_SENTINEL` to distinguish explicit from default, which is robust against future flag additions.

The confidence threshold (`CONFIDENCE_THRESHOLD = 0.60`) is enforced at every call site: CLI `translate`, CLI `run`, GUI Phase 1, and GUI Phase 2 all check `confidence >= CONFIDENCE_THRESHOLD` before applying the detected language, falling back to the default with a warning when below threshold.

The installer `build_installer.py` does not synchronize the version between `pyproject.toml` and `stx_installer.iss`. Both hardcode `1.5.0` independently. ISCC supports `/DMyAppVersion=X.Y.Z` injection, so the orchestrator could read the canonical version from `pyproject.toml` and pass it through.

The `_ISO_TO_SALESFORCE` mapping sends `"pt"` to `"pt_BR"` unconditionally. Since `langdetect` does not distinguish `pt-BR` from `pt-PT`, European Portuguese documents will be silently tagged as Brazilian Portuguese.

## Version hardcoding across artifacts

The `.iss` script defines `#define MyAppVersion "1.5.0"` while `pyproject.toml` has its own `version = "1.5.0"`. The `build_installer.py` orchestrator does not read the version from either source — it simply invokes `ISCC` with no `/D` override. ISCC supports `/DMyAppVersion=X.Y.Z` on the command line, so `build_installer.py` could parse the version from `pyproject.toml` and inject it, ensuring the installer always matches the package version without manual coordination.

## Portuguese mapping bias

The `_ISO_TO_SALESFORCE` dictionary maps `"pt"` to `"pt_BR"`. The `langdetect` library returns `"pt"` for both Brazilian and European Portuguese (it does not distinguish `pt-BR` from `pt-PT` in its output code). This means a document entirely in European Portuguese will be tagged as Brazilian Portuguese in Salesforce. Since the mapping table is the single authoritative source, this is a documentation issue rather than a bug — the choice should be explicitly stated in a code comment so users inspecting the mapping understand the behavior.

## Test coverage observations

The `test_importerror_graceful` test patches `builtins.__import__` directly to simulate `langdetect` being unavailable. This approach is brittle across Python versions (the `__builtins__` object differs depending on whether it's accessed from `__main__` or a module). A more robust alternative would be to patch `stx.lang_detect.detect_langs` at the module level. Since it passes in the current test matrix, this is informational rather than blocking.

There are no tests for caller-side threshold enforcement (i.e., that the CLI or GUI actually falls back when confidence is low). Given the implementation is straightforward conditional logic, this is a low-risk gap.
