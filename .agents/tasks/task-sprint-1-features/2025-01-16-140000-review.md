# Validation report export and row-level approved status

Two features land together: a validation report exporter (CSV/JSON/HTML via CLI and GUI) and an "approved" lifecycle state for translation entries that persists through Excel and STF, skips validation, and gains CLI batch commands. The change is additive — the model gains a field, all writers/readers learn to persist it, and the validator learns to skip it.

Watch for: Phase 5 inline editor drops the `approved` flag on manual edits (confirmed). `render_translated_only_stf` silently strips `# APPROVED` markers on export (confirmed). The `# APPROVED` marker lives in a fifth tab-column which Salesforce may reject on re-import (possible).

## High-level view

The `Entry` dataclass stays frozen and gains `approved: bool = False`. Because it's frozen, every mutation site must reconstruct the full Entry — the autofix functions propagate the field, Phase 4's GUI propagates it, but Phase 5's `_on_apply_edit` omits it, silently clearing approval on any manual edit.

The STF persistence encodes approval as a fifth tab-separated field (`\t# APPROVED`) after the out-of-date column. The parser reads column index 4; the full-STF writer emits it on translated rows. However `render_translated_only_stf` does not emit the marker, so the "TranslatedOnly" export file loses approval data — a lossy asymmetry between the three STF output files.

The approve/unapprove CLI commands load a document from STF or Excel, mutate entries by key or by predicate, and write back via shared `_load_source`/`_save_source` helpers. The validator skips approved entries for per-entry checks but still runs duplicate-key detection on all rows, though the info-level "N entries skipped" message inflates the issue count in the GUI banner, preventing "All clear" from ever appearing when approved rows exist.

## Phase 5 inline editor loses approval on manual edit

In `phase5_validate.py` line 537:

```python
self._state.document.entries[i] = Entry(
    key=e.key, label=e.label, translation=new_translation
)
```

The `approved` kwarg is absent, so the dataclass default (`False`) takes over. Every manual translation edit in Phase 5 silently clears the entry's approved status. Phase 4's equivalent code correctly passes `approved=old.approved` — this is an oversight in Phase 5 only. (confirmed)

## Approval marker lost in TranslatedOnly STF export

`render_full_stf` appends `\t# APPROVED` for approved entries. `render_translated_only_stf` does not:

```python
lines.append(f"{entry.key}\t{entry.label}\t{entry.translation.strip()}\t-")
```

If a user re-imports from the "TranslatedOnly" STF file (a plausible workflow for reviewers who only deal with translated rows), all approval data is silently lost. The asymmetry between the two renderers will surprise users who expect approval state to survive any STF round-trip. (confirmed)

## STF fifth-column interoperability risk

The `# APPROVED` marker occupies a fifth tab-delimited field: `key\tlabel\ttranslation\t-\t# APPROVED`. Standard Salesforce STF files use four columns. If this file is uploaded to Salesforce Translation Workbench or processed by another tool that expects exactly four columns, the marker may cause parse failures or be silently dropped. The risk lives in the external ecosystem, not in this codebase's parser. (possible — would need a Salesforce test import to confirm)

## Info-level "approved_skipped" message and UI counters

`validate_document` appends an info-level issue: `"{N} approved entry(ies) skipped during validation."` Phase 5's banner logic:

```python
if not self._report.issues:
    # "All clear"
```

This path is unreachable when any approved entries exist, because the info issue is always appended. A document with 500 approved rows and zero actual problems shows "1 issue" with an error-styled banner. The counter and banner should filter to `severity != "info"` or the info message should be communicated outside the issues list. (confirmed)

## Test coverage gaps

Not tested: the Phase 5 `_on_download_report` GUI flow, Phase 4 checkbox toggling via the model's `setData`, interaction between approval and auto-fix (the autofix code propagates the field, but no test exercises this end-to-end), and the info-level skip message inflating the UI banner.
