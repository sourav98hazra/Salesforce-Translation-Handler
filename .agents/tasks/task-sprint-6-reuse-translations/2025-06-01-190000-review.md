# Import existing translations and retranslation control

Two features that give users fine-grained control over which translations get reused and which get regenerated. Feature 10 adds an Excel import path that feeds a source‚Üítranslation dictionary into the runner at highest priority (before TM, before network). Feature 11 adds a `retranslate_existing` flag that overrides the default skip-already-translated behavior. Both are wired through CLI flags, GUI settings, and the TranslationWorker.

Watch for:
- **Imported translations bypass retranslate_existing skip gate** (confirmed) ‚Äî when `retranslate_existing=False` and a row already has a translation, that row is skipped in `_classify` before `_translate_one` is ever called, meaning imported translations cannot apply to already-translated rows even if the user loaded an import file expecting to override them.
- **Silent failure on import parse errors during page activation** (confirmed) ‚Äî the phase 3 `_activate` path catches all exceptions and silently passes, giving the user no indication their settings-persisted import file is now invalid.
- **No GUI test coverage** (confirmed) ‚Äî 30 new tests cover the module and runner integration thoroughly, but the Phase 3 and Phase 4 GUI code paths have zero test assertions.

## High-level view

The import translations module is a flat dict lookup that sits at higher priority than the TM in `_translate_one`, populates the dedup cache on first hit, and returns before the TM persist step ‚Äî meaning imported translations don't write back to the TM and don't persist beyond the current run.

The retranslation control changes the `_classify` gate: when true, rows with existing translations proceed to `_translate_one` rather than being skipped. The interaction between the two features has a subtle ordering gap ‚Äî imported translations can only apply to rows that pass through `_classify` into `_translate_one`, so rows skipped by the default `retranslate_existing=False` path are invisible to the import dictionary even if the import file has a match for them.

The GUI integration adds an import button and retranslation checkbox to Phase 3, a context menu to Phase 4 for clearing individual rows, and a settings dialog section for persisting the import file path. Error handling on the settings-restore path is a bare `except: pass`.

## Import priority vs. the skip gate in `_classify`

The runner's two-pass architecture (classify then translate) creates an ordering dependency. In `_classify`, the existing-translation check fires before `_translate_one` is ever called:

```python
if entry.translation.strip():
    if self.retranslate_existing:
        return "translate"
    self._fill(index, entry, sheet, "Skipped (already translated)", summary)
    ...
    return "done"
```

The imported translations lookup lives inside `_translate_one`. So Feature 10's "highest priority" claim only holds for rows that are either untranslated or explicitly opted into retranslation. A user who imports a corrected Excel expecting it to silently patch already-translated rows will find those rows unchanged unless they also check the retranslation flag. The test `test_imported_priority_without_retranslate` confirms this behavior (k2 keeps its existing translation) but doesn't test the case where the user *has* an imported translation for an already-translated row with `retranslate_existing=False`.

The fix could be a third check in `_classify`: if the imported dict has a hit for this label, let it through to `_translate_one` regardless of existing translation state.

## Phase 3 activation: silent failure path

```python
if import_path and import_enabled and Path(import_path).exists():
    from ...import_translations import parse_translation_file
    try:
        result = parse_translation_file(Path(import_path))
        ...
    except Exception:
        pass
```

The `Path(import_path).exists()` check prevents the most common failure (file moved), but if the file exists with corrupted content, the user gets no feedback. The label remains empty, the state remains `None`, and the translation run proceeds without imported translations. A log statement or a brief status label like "Import file could not be read" would surface the issue.

## Double-counting in retranslated_count

In `_fill_translated`:

```python
if self.retranslate_existing and entry.translation.strip():
    self._retranslated += 1
```

This fires regardless of *how* the row was translated (imported, TM, dedup, network). A row that had an existing translation, passed through because `retranslate_existing=True`, and then matched an imported translation gets counted in both `_imported_reuse` and `_retranslated`. The CLI summary renders something like:

```
Translated 10 (TM hits 2, fuzzy 0, dedup 1, imported 3, retranslated 5, resumed 0); skipped 2
```

The tests assert this double-count explicitly (`test_imported_takes_priority` expects `retranslated_count == 3` when all 3 have existing translations and 2 come from imports). Users who read "retranslated" as "sent to the network translator again" may find the numbers confusing.
