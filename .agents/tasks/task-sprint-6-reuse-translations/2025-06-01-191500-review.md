# Import existing translations and retranslation control (v2)

Two features giving users fine-grained control over which translations get reused and which get regenerated. Feature 10 adds an Excel import path feeding a source→translation dictionary into the runner at highest priority. Feature 11 adds a `retranslate_existing` flag overriding the default skip-already-translated behavior. Both are wired through CLI, GUI, and TranslationWorker.

All four issues from the v1 review have been addressed: (1) imported translations now bypass the skip gate in `_classify` regardless of `retranslate_existing`, (2) parse failures during phase 3 activation now log a warning and show a user-visible error label, (3) `from_imported` rows are excluded from `retranslated_count`, and (4) a new test (`test_imported_overrides_already_translated_without_retranslate`) explicitly covers the bypass scenario.

Watch for:
- **Checkpoint trumps import on resume** (likely) — a checkpointed row is restored before the import-bypass check runs, so a resumed session won't apply newly-imported translations to rows that were already checkpointed in a prior run.
- **No GUI test coverage** (confirmed) — the Phase 3 import button, retranslation checkbox, Phase 4 context menu, and settings dialog import section remain untested.

**Verdict**: APPROVED

## High-level view

The import bypass in `_classify` is correctly ordered: checkpoint → import dict check → existing-translation gate. Imported translations override already-translated rows without requiring the retranslate flag. The only case where an imported translation is ignored is when a checkpoint entry exists for that index — a narrow scenario affecting only resumed sessions where the user adds an import file after a prior run partially completed.

## Checkpoint vs. import priority ordering

In `_classify`, the decision pipeline runs: cancel → scope → **checkpoint resume** → **import bypass** → existing-translation gate. Because checkpoint fires first, a row that was translated and checkpointed in a previous run is unconditionally restored from checkpoint data:

```python
if index in self._checkpoint_data:
    cp = self._checkpoint_data[index]
    translation = cp.get("translation", "")
    ...
    return "done"

# Import bypass: if the imported translations dict has a match...
if self.imported_translations and entry.label in self.imported_translations:
    return "translate"
```

A user who resumes a partially-completed translation and simultaneously adds an import file expecting it to patch previously-checkpointed rows will find those rows unchanged. This only matters when both `--resume` and `--import-translations` are active and the import file has better translations for rows that were already processed. The checkpoint is cleared on successful completion, so a fresh run after completion would pick up the import file correctly.

## Test gaps

Not tested:
- GUI widget behavior (button visibility, checkbox state, file dialog flow, settings persistence round-trip, context menu appearance)
- CLI end-to-end with `--import-translations` and `--retranslate-existing` flags combined
- Error paths in the GUI (corrupted import file, permission errors on the button handler path)
- Multi-target translation with imported translations (`translate_document_multi` passes the dict through but has no dedicated test)

The module-level and runner-integration tests (31 total across two files) cover the core logic thoroughly, including the bypass scenario, dedup propagation, priority over TM, and the retranslated_count semantics.
